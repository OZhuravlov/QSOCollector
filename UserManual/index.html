🎯 Overview

QSOCollector is a professional-grade Windows application designed specifically for multi-station DXpeditions to collect, manage, and export QSO (radio contact) records with reliability and precision.

What is QSOCollector?

QSOCollector solves critical operational challenges faced by DXpedition teams:

  • 🎯 Centralized Database - Single source of truth for all QSO records
  • 🔒 Reliable Collection - TCP-based transmission eliminates QSO loss in WiFi networks
  • 📊 Duplicate Prevention - Intelligent filtering prevents re-export of same QSOs
  • 🌐 Multi-Station Support - Collects from unlimited logging stations
  • 📁 Format Flexibility - Supports N1MM and ADIF formats
  • ⚡ Network Resilience - Automatic local caching if server unavailable

Key Capabilities

Reliable QSO Collection TCP protocol ensures no QSO loss, automatic retries, local buffering
Centralized Storage All QSOs stored in central SQLite database on server PC
Network Tolerance Continues collecting when server unavailable, auto-syncs when restored
Flexible Export Export new, by date range, or all QSOs with filtering options
QSO Search Quickly find QSOs by callsign, band, mode with sorting and filtering
Automatic Startup Runs at Windows startup, minimizes to tray, always collecting

Who Should Use QSOCollector?

  • 🏕️ DXpedition Teams - Multiple stations, multiple PCs
  • 📡 Contest Clubs - Centralized QSO management
  • 🏛️ QSL Bureaus - Import/export ADIF archives
  • 🔬 Radio Researchers - Analyze propagation patterns
💡 Tip: For best results, ensure QSOCollector runs on all logging PCs before loggers start. This guarantees no QSOs are lost at DXpedition start.

📥 Installation

Step 1: Download QSOCollector

  1. Visit GitHub Releases Page
  2. Download latest release MSI file
  3. If you have a previous QSOCollector version installed then installer will prompt to uninstall it
  4. It's recommended to uninstall a previous version manually before installing a new one. See "Uninstallation" instructions bellow
  5. If you don't have .NET 10 Runtime installed then installer will prompt to install it. Otherwise skip Step 2

Step 2: Install .NET 10 Runtime

  1. Download from Microsoft .NET Download
  2. Choose "Desktop Runtime" (not SDK or ASP.NET)
  3. Run installer and follow prompts

Step 3: Verify Installation

  1. Double-click QSOCollector.exe
  2. Application window should open
  3. Check both "Server" and "Client" tabs are visible
  4. Database created automatically in: 
    C:\Users\[YourUsername]\Documents\QSOCollector\db\

Step 4: Configure Windows Firewall

For Windows Firewall to allow QSOCollector:

  1. Open Windows Defender Firewall
  2. Click "Advanced settings"
  3. Right-click "Inbound Rules" → "New Rule..."
  4. Select "Program" radio button
  5. Click "Browse..." and select QSOCollector.exe
  6. Select "Allow the connection"
  7. Check all three: Domain, Private, Public
  8. Enter name: "QSOCollector" and finish

Step 5: First Launch

  • ✅ Application opens with default settings
  • ✅ Enable "Server" and/or "Client" mode
  • ✅ Configure TCP/UDP ports
  • ✅ Click "Start Server" and/or "Start Client"

Uninstallation

Remove QSOCollector as regular Windows application:

  1. Close application completely
  2. Go to Windows Settings -> Apps -> Installed apps
  3. Find QSOCollector in the app list and uninstall it
  4. If you are not going to use QSOCollector anymore then
  5. delete the following folder manually: C:\Users\[YourUsername]\Documents\QSOCollector
  6. ! Be aware this will remove QSO database with no further recovery possible
  7. (Optional) Remove Windows Firewall rule
💾 Important for Server: Export all QSO records to ADIF file before deleting C:\Users\[YourUsername]\Documents\QSOCollector folder.

Network Setup & Configuration

Finding Your Network Information

You'll need to know your PC's IP address and available network ports.

Find Your PC's IP Address

  1. Open Command Prompt (WIN+R, type cmd)
  2. Type: ipconfig
  3. Look for "IPv4 Address" (e.g., 192.168.1.50)
  4. This is your Server PC IP

Note: IP addresses starting with 192.168 or 10. are private (local network only)

Check Port Availability

netstat -ano | findstr :5000

If no output appears, port 5000 is available. Try different ports if needed.

Client Configuration

Client Mode Overview

Client mode runs on each PC with loggers and collects QSOs via UDP, then sends to Server via TCP.

  • ✅ Multiple Clients allowed (one per logging PC)
  • ✅ Listens for UDP broadcasts from multiple local loggers
  • ✅ Sends QSOs to Server via TCP
  • ✅ Auto-buffers QSOs if Server unavailable
  • ✅ Must run whenever logger is operating
⚠️ Warning:QSOs will not be collected if Client is not running. Always start Client before starting loggers to ensure no QSOs are lost.

Basic Client Setup

  1. Launch QSOCollector.exe on logger PC
  2. Click "Client" tab
  3. Enter:
    • Server IP/Hostname: Address of server PC (e.g., 192.168.1.50)
    • Server Port: Must match server (e.g., 5000)
  4. Click "UDP Listeners" to configure
  5. Click "Start Client"

Client Status Indicators

Status Color Meaning Action
Connected 🟢 Green Connected to Server Normal operation - QSOs send immediately
Disconnected 🔴 Red Server unavailable QSOs buffered locally - will sync when available
Unknown 🟡 Lavender Haven't tried to connect Will try when Client starts

Server Connection Settings

  • Server IP: Server local network IP address like example: 192.168.1.50
  • Server Port: MUST match server configuration (default 5000)
  • Timeout: After 30 seconds no response, switches to offline mode
  • Offline mode: QSOs saved in local database when server is unavailable
  • Online mode: New QSOs sent directly to the server, saved QSOs sent once connection restored

UDP Listeners Configuration

What are UDP Listeners?

UDP Listeners collect QSO broadcasts from loggers on the local PC via UDP protocol. Each PC can have multiple loggers, each sending to different UDP ports.

Accessing UDP Listeners

  1. Click "Client" tab
  2. Click "UDP Listeners" button
  3. UDP Listeners dialog opens

Creating a Listener

  1. In UDP Listeners form, click on the last empty row
  2. New listener line appears in list
  3. Edit each field (see required fields below)
  4. Click "Save" to persist configuration

UDP Listener Fields

Field Required Example Description
Listener Name ✅ Yes N1MM Logger Any unique name to identify this listener
QSO Port ✅ Yes 12060 Port logger broadcasts on (from logger config), unique
Forward Port ❌ No 12080 forward QSOs to other apps (e.g., ClubLog)
Heart Beat Port ❌ No 12061 Optional - listen for keep-alive messages from logger
Format ✅ Yes N1MM or ADIF Format of QSO data from logger
Active ✅ Yes Checked Enable/disable without removing listener

Forward Port Usage

Why use Forward Port?

On Windows, only one application can listen on a specific UDP port. If another application (e.g., ClubLog Livestream) needs QSO data:

  1. Logger broadcasts to port 12060
  2. QSOCollector listens on 12060
  3. QSOCollector forwards to port 12080
  4. ClubLog Livestream listens on 12080
  • Forwat port could be the same for different Listeners with the same QSO format

Managing Listeners

  • Add: Click on the last (empty) row to start adding new listener
  • Remove: Select listener, click "Remove"
  • Edit: Double-click field to edit
  • Enable/Disable: Check/uncheck "Active" checkbox
  • Save: Click "Save" to persist all changes

Importing/Exporting Listener Configuration

  1. In UDP Listeners dialog, click "Export Config"
  2. Saves configuration to JSON file
  3. On another PC, click "Import Config"
  4. Select the JSON file
  5. Configuration loaded and applied

Example config file:

[
  {
    "Id": 40,
    "Name": "MSHV",
    "QsoPort": 2233,
    "ForwardPort": 2244,
    "AcknowledgePort": 2237,
    "MessageFormat": "ADIF",
    "IsActive": true
  },
  {
    "Id": 42,
    "Name": "DXLOG",
    "QsoPort": 12050,
    "ForwardPort": 12061,
    "AcknowledgePort": 12052,
    "MessageFormat": "N1MM",
    "IsActive": true
  }
]

Troubleshooting Listeners

Problem: No QSOs received when put into logger

  • Verify brodcasting configured and enabled in logger
  • Verify QSO Port matches logger configuration
  • Verify Format matches logger output (N1MM or ADIF)
  • Check firewall isn't blocking UDP port
💡 Tip: Save your listener configuration by exporting to file. This makes setup much faster on multiple PCs.

What If Server Becomes Unavailable?

Automatic Offline Handling:

  1. Server Status changes to 🔴 Unavailable
  2. QSOs from logger continue to be received
  3. QSOs automatically saved to local database
  4. "Temporary Saved" counter increases
  5. Client continues retrying connection automatically

When Server Comes Back Online:

  1. Client automatically reconnects
  2. Status changes to 🟢 Connected
  3. All buffered QSOs automatically sent to server
  4. "Temporary Saved" counter decreases
  5. Normal operation resumes
✅ Zero Data Loss: No QSOs are lost even if server is unavailable for hours or days. They're cached locally and transmitted when connection restored.

Monitoring Client Statistics

Statistic Meaning Normal Value
Received QSOs received from local logger(s) Increases during operation
Sent to Server QSOs successfully sent (when connected) ~= Received (when connected)
Temporary Saved QSOs buffered locally (when disconnected) 0 when connected, increases when offline
Rejected Invalid QSOs (format errors) Should be 0 (investigate if > 0)

Diagnosing Client Issues

Symptom: Server Status stays 🔴 Unavailable

  • Check Server IP address is correct
  • Check Server Port matches server
  • Ping server: ping [server-ip]
  • Test port connectivity: Test-NetConnection -ComputerName [ip] -Port [port]
  • Check Windows Firewall on both PCs
  • Check router firewall/port filtering

Symptom: QSOs not arriving at server

  • Check UDP Listeners are configured correctly
  • Verify UDP listener "Active" checkbox is checked
  • Verify logger port matches QSO Port in listener
  • Verify Format (N1MM or ADIF) matches logger output
  • Enable "Log details" to see UDP reception events

Symptom: "Temporary Saved" count increasing but not clearing

  • Server is unavailable or connection broken
  • Check server is actually running
  • Check network connection (test ping)
  • Restart client to force reconnection attempt

Stopping the Client

  1. Click "Stop Client" button
  2. UDP listeners stop
  3. Connection to server closes gracefully
  4. Any buffered QSOs stay in local database for next startup
⚠️ QSO Loss Warning: Don't stop client while operating. Any QSO put into logger when Client stopped won't be collected by QSOCollector

Autostart Configuration

To ensure client never misses QSOs:

  1. Check "Start automatically" checkbox
  2. Client will start with Windows automatically
  3. Minimized to tray (can double-click tray icon to restore)
  4. All Configuration retained from previous session
💡 Best Practice: Always enable "Start automatically" on client PCs so QSOs are captured even if PC is rebooted.

Export Overview

Export QSOs from database to ADIF format for uploading to ClubLog or other services.

Export Steps

  3. Export dialog opens
  4. Select filter (see below)
  5. Configure secondary filters (optional)
  6. Click "Export"
  7. Choose filename and location
  8. Review export results
  9. Mark as exported (recommended)

Export Filter Options

Filter When to Use Example
"New QSOs" Daily export to ClubLog (most common) Export all QSOs since yesterday
"By Date" Specific date range Export Feb 10-12 for special event
"All QSOs" Full database export Export everything for backup or before totally removing QSOCollector Server from PC

Using "New QSOs" Filter (Recommended)

  1. Click "QSO Export"
  2. Select "New QSOs" radio button
  3. (Optional) Apply secondary filters (Mode Group, Band, Operator)
  4. Bottom shows: "Filtered: X QSOs" (number to export)
  5. Click "Export"
  6. Choose filename (default example: export_20260213_100215.adi)
  7. Click "Save"
  8. Results shown (e.g., "Exported 245 QSOs")
  9. Recommended: Click "Mark as exported" checkbox to prevent future re-export
  10. ADIF file opened in Explorer for upload to any other applications, i.e. ClubLog

Secondary Filters

After selecting main filter, apply optional secondary filters:

  • Mode Group: CW, SSB, DATA, etc. (narrow to specific mode)
  • Band: 80m, 40m, 20m, etc. (narrow to specific band)
  • Operator: Specific operator name

Secondary filters apply IN ADDITION TO main filter.

Export Workflow for Daily ClubLog Upload

  1. Open QSOCollector Server
  2. Click "QSO Export"
  3. Select "New QSOs"
  4. Click "Export"
  5. Use default filename (includes date/time)
  6. Check "Mark as exported" ← IMPORTANT!
  7. Upload resulting ADIF file to ClubLog
  8. Verify in ClubLog
  9. Next day, repeat → only new QSOs will export

  • Import default: %UserProfile%\Documents\QSOCollector\import
  • Folders created automatically on first use

ADIF Format Support

  • ✅ Supports standard ADIF files
  • ✅ Handles common variations
  • ✅ Validates QSO data
  • ⚠️ Invalid QSOs skipped with warning
  • ❌ Non-ADIF formats not supported

Troubleshooting Import/Export

Import fails with "Invalid ADIF file"

  • Verify file is actually ADIF format
  • Open in text editor - should contain ADIF tags (<call>, <qso_date>, etc.)
  • Try opening file in another ADIF reader first
  • Check file not corrupted

Export creates empty file

  • Verify "Filtered" count shows QSOs at bottom
  • May have selected date range with no QSOs
  • Try "All QSOs" filter instead

QSOs exported more then one time

  • Did NOT mark QSOs as exported during previous export
  • Solution: Mark as exported
💡 Tip: Save all export ADIF files with descriptive names including dates. Useful for backup and audit trail.

QSO Search Feature

Overview

Search for specific QSO records in the database by callsign, band, or mode. Perfect for verifying QSOs or finding duplicate records.

Opening QSO Search

  1. Click "Server" tab
  2. Click "Search QSOs" button
  3. QSO Search modal dialog opens

Basic Search

  1. Enter callsign in search box (e.g., "N0CALL")
  2. Minimum 3 alphanumeric characters required
  3. Text auto-converts to UPPERCASE
  4. Press Enter or click "Search" button
  5. Results appear in grid (max 200)

Search Patterns & Wildcards

Pattern Example Result
Simple callsign N0CALL callsing is exactly "N0CALL"
Start with NC0% Starts with "NC0" (NC0D, NC0XY, etc.)
End with %CALL Ends with "CALL" (N0CALL, W5CALL, etc.)
One char wildcard N_CALL N + any char + CALL (N0CALL, NXCALL, etc.)
💡 Tip: Use % (percent) wildcard liberally. "NC0%" finds all NC0 stations, etc.

Using Filters

  1. After search results appear, filters are auto-populated
  2. Mode Group: Select CW, SSB, DATA, etc.
  3. Band: Select 80m, 40m, 20m, etc.
  4. Results filtered to match criteria
  5. Can combine both filters

Results Table

Results display in a table with columns:

  • Callsign: QSO partner's call
  • QSO Time: Date and time (YYYY-MM-DD HH:MM:SS)
  • Mode Group: CW, SSB, DATA (category)
  • Mode: Specific mode (CW, USB, PSK31, etc.)
  • Band: Frequency band (80m, 40m, etc.)

Network Architecture & Advanced Setup

Network Communication Model

QSOCollector uses two types of network communication:

  • UDP (Local): Logger → Client (same PC, no network)
  • TCP (Network): Client → Server (over LAN/WiFi)

Network Flow Diagram

    ┌─────────────────────────────────────────────────────┐
    │  Server PC (192.168.1.50)                           │
    │  ┌──────────────────────────────────────────────┐  │
    │  │ QSOCollector Server                          │  │
    │  │ • TCP Listen: 0.0.0.0:5000                  │  │
    │  │ • Central SQLite Database                    │  │
    │  └──────────────────────────────────────────────┘  │
    │           ↑              ↑              ↑            │
    │           │              │              │            │
    │      TCP  │         TCP  │         TCP  │            │
    │      5000 │         5000 │         5000 │            │
    │           │              │              │            │
    │  Client PC #1      Client PC #2    Client PC #3     │
    │  (192.168.1.51)    (192.168.1.52)  (192.168.1.53)   │
    │      ↑                  ↑               ↑             │
    │      │ UDP              │ UDP           │ UDP         │
    │      │ 12060            │ 10001         │ 12060       │

Network Reliability Features

TCP Advantages over UDP:

  • ✅ Guaranteed delivery: TCP ensures QSOs arrive in order
  • ✅ Automatic retransmission: Failed packets resent automatically
  • ✅ No duplication: TCP handles duplicate elimination
  • ✅ Connection-based: Clear state (connected/disconnected)
  • ✅ Flow control: Sender adapts to receiver speed

UDP (logger-to-client) is acceptable because:

  • Local network (same PC or LAN segment) is reliable
  • Client can retry QSOs locally
  • Lower latency than TCP
  • Logger broadcast standard for compatibility

Offline Caching Mechanism

When server unavailable, client automatically buffers QSOs:

  1. Client receives QSO from logger (via UDP)
  2. Attempts to send to server (TCP)
  3. If TCP fails, saves to local SQLite database
  4. Continues receiving and buffering QSOs
  5. When server comes back online, resends all buffered QSOs
  6. Clears buffer once confirmed by server

Bandwidth Requirements

  • Per QSO: ~150 bytes typical
  • 1000 QSOs/hour: ~150 KB/hour (~0.042 MB/sec)
  • Very low bandwidth: Works on any home WiFi or 4G
  • Typical DXpedition: <1 MB/day per PC

Latency Tolerance

  • Typical LAN: <5ms latency (imperceptible)
  • WiFi: 5-50ms (acceptable)
  • Internet WAN: 50-200ms (still works)
  • Buffer timeout: 30 seconds (very tolerant)

Multi-PC Scalability

  • Server handles unlimited client connections
  • Typical DXpedition: 5-10 client PCs
  • Load tested up to 50+ concurrent connections
  • CPU usage minimal (<5% per client on typical server)

Advanced: DHCP vs Static IP

Recommended: Static IP for Server PC

  • Ensures clients always find server
  • Prevents connection issues if IP changes

How to set static IP on Windows:

  1. Settings → Network → Advanced network settings
  2. IP settings → Edit
  3. Select "Manual" → IPv4
  4. Enter IP (e.g., 192.168.1.50)
  5. Enter Gateway (usually 192.168.1.1)
  6. Enter DNS (e.g., 8.8.8.8)

Alternative: DHCP Reservation

  • Configure router to always assign same IP to server PC
  • Easier than manual static IP
  • Check your router manual for how to set DHCP reservation

Automatic:

Autostart & System Tray

Why Enable Autostart?

Critical for DXpeditions: QSOCollector Client MUST be running when loggers start operating, or QSOs will be lost!

  • ✅ Ensures no QSOs missed if PC rebooted
  • ✅ Application starts automatically with Windows
  • ✅ Runs with previous configuration
  • ✅ Minimized to tray (not obtrusive)

Enabling Autostart

  1. In QSOCollector, check "Start automatically" checkbox (top-right)
  2. Application will now:
    • Start automatically when Windows boots
    • Load previous configuration
    • Minimize to system tray (not visible on desktop)
    • Continue operation in background

System Tray Integration

When minimized to tray:

  • Application icon appears in system tray (bottom-right)
  • Double-click tray icon to restore window
  • Right-click tray icon for menu options
  • Application continues collecting QSOs in background

Disabling Autostart (Not Recommended)

  1. Uncheck "Start automatically" checkbox
  2. Application will no longer auto-start
  3. Must be launched manually next time

Advanced: Manual Autostart Configuration

If checkbox doesn't work, configure Windows Task Scheduler:

  1. Open Task Scheduler (search in Windows)
  2. Click "Create Basic Task"
  3. Name: "QSOCollector"
  4. Trigger: "At log on"
  5. Action: "Start a program"
  6. Program: Browse to QSOCollector.exe
  7. Click "Finish"
  8. Test: Restart PC, verify QSOCollector starts

Startup Behavior

On first startup after enabling autostart:

  • Application loads current settings
  • Server or Client mode starts based on last configuration
  • Connection attempts begin automatically
  • Window minimized to tray
  • All activity continues in background
💡 Tip: For server PC, enable autostart so server starts automatically after power loss. For client PCs, enable to ensure QSOs always captured.

Troubleshooting Autostart

Application not autostarting:

  • Verify "Start automatically" checkbox is checked
  • Try Task Scheduler method (see above)
  • Check Windows startup programs (Settings → Apps → Startup)
  • Verify .NET 10 runtime installed (required for startup)

Want to prevent autostart temporarily:

  • Uncheck "Start automatically" before shutdown
  • Or disable in Task Scheduler
  • Re-enable after maintenance

Troubleshooting Guide

Common Issues & Solutions

Server Won't Start

Symptoms: Click "Start Server" but nothing happens, or error message appears

Solutions:

  1. Check if port 5000 is already in use: 
    netstat -ano | findstr :5000
  2. If port in use, try different port (5001, 5002, etc.)
  3. Check Windows Firewall allows QSOCollector
  4. Check disk space (at least 100 MB free)
  5. Verify .NET 10 runtime installed: 
    dotnet --version
  6. Restart application
  7. Restart Windows

Client Shows "Disconnected" / Server Unavailable

Symptoms: Client status shows 🔴 Disconnected even though server running

Solutions:

  1. Verify Server IP address is correct (don't use "localhost")
  2. Verify Server Port matches server configuration
  3. Test connectivity from client PC: 
    ping [server-ip]
  4. Test port connectivity: 
    Test-NetConnection -ComputerName [server-ip] -Port 5000
  5. Check Windows Firewall on both PCs
  6. Check router allows traffic on port 5000
  7. Restart both server and client

QSOs Not Arriving at Server

Symptoms: Logger running, client shows "Connected" but no QSOs in server

Solutions:

  1. Verify UDP Listeners are configured:
    • Listener Name: Any name
    • QSO Port: Must match logger broadcast port
    • Format: Must match logger format (N1MM or ADIF)
    • Active: Must be checked ✓
  2. Check logger is broadcasting (verify in logger settings)
  3. Enable "Log details" on client to see UDP reception
  4. Test logger UDP broadcast directly: 
    netstat -ano | findstr UDP
  5. Restart logger
  6. Restart client

Duplicate QSOs in ClubLog

Symptoms: Same QSOs appearing twice in ClubLog after export/upload

Solutions:

  1. Most likely cause: Did NOT mark QSOs as exported after previous upload
  2. When exporting, ALWAYS check "Mark as exported" checkbox before upload
  3. Use "New QSOs" filter for daily exports (prevents re-exporting)
  4. To fix existing duplicates in ClubLog, contact ClubLog support
  5. Next time, mark after each export to prevent repeat

High Memory or CPU Usage

Symptoms: Application uses lots of resources, PC sluggish

Solutions:

  1. Disable "Log details" checkbox (reduces verbose logging)
  2. Check database size isn't huge (database grows over time)
  3. Reduce number of connected clients if many
  4. Restart application to clear memory
  5. Ensure no multiple instances running
  6. Monitor Task Manager to verify it's QSOCollector using resources

Database Corruption Error

Symptoms: Error messages about "database locked", "disk I/O error", or similar

Solutions:

  1. Close QSOCollector application completely
  2. Wait 10 seconds
  3. Reopen QSOCollector
  4. If error persists:
    1. Backup database file from C:\Users\[YourUsername]\Documents\QSOCollector\db\qsoCollector.s3db
    2. Delete qsoCollector.s3db
    3. Reopen QSOCollector (new database created)
    4. Import from backup ADIF file if needed

Import/Export Errors

Symptom: "Invalid ADIF file" error during import

  • Verify file is actually ADIF format
  • Open in text editor - should contain ADIF tags (<call>, <qso_date>)
  • Check file not corrupted or truncated
  • Try opening in another ADIF reader first

Symptom: Export creates empty or very small file

  • Check "Filtered" count at bottom of export dialog
  • If 0, try different filter (All QSOs, different date range)
  • Verify database has QSOs (check Server stats)

QSO Search Not Working

Symptom: Search button disabled or no results

  • Ensure at least 3 alphanumeric characters entered
  • Invalid characters rejected - only A-Z0-9/_%$ allowed
  • Click "Search" button or press Enter
  • If no results, try wider pattern (e.g., "N0%" instead of "N0CALL")

Frequently Asked Questions

General Questions

Q: Is QSOCollector free?

A: Yes! QSOCollector is open-source and free to use. Source code available on GitHub.

Q: Can I run multiple servers?

A: No. Only ONE server per DXpedition. All clients must connect to the same server. Multiple servers would cause database conflicts.

Q: Can I upgrade from version X to Y without losing QSOs?

A: Yes. Database is backward compatible. QSOs are preserved during upgrade. Backup before upgrading (always good practice).

Q: Does QSOCollector work on Mac/Linux?

A: Currently Windows only (requires Windows Forms). Cross-platform support would require major refactor.

Q: Can I use QSOCollector for contests (not just DXpeditions)?

A: No. There is no option to collect and export QSO data in Cabrilo format

Setup & Configuration

Q: How do I find my PC's IP address?

A: Open Command Prompt, type ipconfig, look for "IPv4 Address" (typically starts with 192.168 or 10.)

Q: What if Server PC's IP address changes?

A: Clients will lose connection. Set Server PC to static IP (not DHCP) or use DHCP reservation. See Network Setup section.

Q: Can I access QSOCollector from another building/location?

A: Only if connected via VPN. Do NOT expose port 5000 directly to Internet (security risk). Use VPN for secure remote access.

Q: What port numbers should I use?

A: Recommended: Server TCP 5000, UDP Logger ports 12000+. Any available ports work. Check with netstat.

Operation & QSOs

Q: QSOCollector running but logger not connecting. What's wrong?

A: Verify in logger settings that it's broadcasting on correct port. Check QSOCollector UDP Listener configured for that port. Enable verbose logging to debug.

Q: Can I mix N1MM and ADIF loggers on same network?

A: Yes! Configure multiple UDP Listeners - one set to N1MM format, another to ADIF. They coexist fine.

Q: What if logger sends duplicate QSOs?

A: QSOCollector detects duplicates by call/time and rejects them. Database won't have duplicate entries.

Q: How often should I export QSOs?

A: Daily is recommended. You can also enable QSO auto exporting by defined scheduler.

Q: Can I edit QSOs in the database?

A: Not directly through QSOCollector UI. If errors found, delete incorrect QSO by exporting to ADIF, editing, then re-importing. Better: verify at logger stage.

Troubleshooting

Q: Server running but clients can't connect. Firewall issue?

A: Possibly. Add QSOCollector.exe to Windows Firewall exceptions. Also check router firewall if applicable. Test with Test-NetConnection.

Q: How do I know if QSOs are being received?

A: Check Server Log for "Received QSO" messages. Check QSO statistics (should increase). Enable "Log details" for more verbose output.

Q: QSOs appearing in database but wrong time zone?

A: QSOCollector always uses UTC time.

Q: Database growing very large, performance degrading. What to do?

A: Export QSOs to ADIF for archival. Cleanup old QSOs from active server database.

Q: Connection keeps dropping between server and client.

A: Check network stability. WiFi issues? Try Ethernet if possible. Enable verbose logging to see exact disconnect times. May be IP address changing (use static IP).

Best Practices

Q: What's the best way to set up for a major DXpedition?

A:

  1. Set up server PC well before operation starts
  2. Test with loggers on each client PC
  3. Verify all clients connecting and QSOs arriving
  4. Enable autostart on all PCs
  5. Set up daily export schedule
  6. Back up database daily

Q: Should I enable "Log details" all the time?

A: No. Only when troubleshooting. It increases CPU/disk usage. Disable for normal operation.

Q: How often should I back up the database?

A: Daily at minimum. After major operations (import/export). Before upgrades. Database is critical - don't lose it!

Q: Can I run server and client on same PC?

A: Yes, but not recommended. Server needs to be always available and reliable. Client may have logger failures. Better to separate.

Q: After DXpedition ends, how long to keep the database?

A: Export to ADIF file and archive. Keep database backup for ~6 months in case issues found. Then can delete to free space.
